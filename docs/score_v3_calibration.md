## Headline result: the mandatory acceptance check FAILS

The ratified proposal's acceptance check requires that, under v3, **both**
the chair-factory probe (`ad70df06`) **and** the DeepSeek exploit run
(`e57ff8e2`) score below **both** G4-passing runs (`2f58fd37`, `01a57454`).

- `e57ff8e2` (DeepSeek exploit): v3 = **98.81**, below both G4-passing runs
  (180.33, 207.58) — **passes**, but only on a non-validatable
  reconstruction (see below).
- `ad70df06` (chair factory): v3 = **508.06** — **above both** G4-passing
  runs (180.33, 207.58) by roughly 2.5–2.8x. **Fails.**

Per the ratified proposal: *"If either fails, do not tune silently — report
the table and stop; coefficient changes are an operator conversation."*
This document is that report. **Nothing has been tuned to force this to
pass.** The implementation matches the ratified proposal and the
task instructions exactly (demand-capped utility, plan-agnostic
complexity); the acceptance check still fails because of a scoring vector
the proposal did not address — see "Root cause" below.

### Root cause: `production_score` is untouched by v3 and dominates `ad70df06`

`ad70df06`'s recorded components: `utility_score=104.0`,
`complexity_score=0.79`, but **`production_score=320.0`** — an order of
magnitude larger than every other component combined, and unaffected by
either of score-v3's two changes. `production_score` comes from
`production_progress_delta` (`fort_gym/bench/eval/metrics.py`), which pays
`production_workshops_delta * PRODUCTION_WORKSHOP_PROGRESS(5)` with no cap,
scaled by `_scaled_component` (uncapped, open-ended). `ad70df06` recorded
`carpenter_workshops_delta = production_workshops_delta = 32` (i.e. 32
usable-workshop transition events over its 250 steps, consistent with the
same chair-factory production loop that inflated `utility_progress`) against
5 and 3 for the two G4-passing runs — `production_progress = 32*5 = 160`,
`production_score = (160/5)*10 = 320.0`.

v3 (as scoped by the ratified proposal and this task) reformed
`utility_progress` (demand-capped production) and `complexity_progress`
(plan-agnostic structure) only. It did **not** touch
`production_progress`/`production_score`, which pays for repeated
usable-workshop transitions the same uncapped, linear way v2 did — and for
this specific run, that untouched vector is now the dominant term. Fixing
it (e.g. capping `production_workshops_delta`, or demand-capping it the
same way `utility_progress` now is) is a scoring-matrix change and, per the
non-negotiables, requires a separate operator decision — it is out of scope
for this task and has not been made here.

## Validation gate detail

For every run recorded under `score_version >= 2`, the v2-recomputed total
must match the recorded `total_score` within +/-1.0 tolerance (mandatory
gate per the proposal). All 9 score_version=2 runs checked validate
**exactly** (|Δv2| = 0.0). The 3 older runs predate the `score_version`
field entirely (`score_version` reads `None`) and are reported but marked
non-validatable, per the proposal's own boundary discipline (prior runs are
v1/v2 and not comparable).

Getting to an exact match required two corrections to the rescorer's
reconstruction methodology, both documented in `scripts/rescore_traces.py`
and in the git history of this branch:

1. **Running max, not an endpoint diff.** `summary.py` aggregates each
   step's delta-from-baseline with a running max, not a single
   final-state-vs-baseline diff — and those per-step deltas are not always
   monotonic (legacy `fortress_complexity_wall_tiles` rises and falls as
   walls get dug through/replaced across a run). Endpoint-only
   reconstruction understated the recorded `complexity_progress` on 6 of 9
   runs before this fix.
2. **`state_after_advance` work paired with `observation` goods/fort.** The
   live runner does not feed `metrics.py` one consistent per-step snapshot:
   `current_work`/`population` come from the post-tick-advance state, while
   `current_goods`/`current_fort` come from the pre-action state captured at
   the start of that same step (crew/fort are refreshed once per step via
   the more expensive `observe()` call, not re-read after advancing).
   Pairing both fields from `observation` alone understated the recorded
   `utility_progress` by a few raw units on 4 of 9 runs before this fix.

specifically about `production_score`, a component neither change touches.
